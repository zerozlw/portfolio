## Background

### The Problem

The Parts Full Lifecycle Platform (IDP) is Xiaomi EV's core parts management system, covering DVP (Design Verification Plan), EA (Engineering Approval), TKO (Technical Kick-Off) and other critical business processes. With 800+ users, the platform faces a growing challenge: **massive amounts of repetitive operational inquiries consume the product and engineering teams' time every week**.

- Product team: **1.5-2 days/week** on maintenance
- Engineering team: **0.5-1 day/week** on debugging

The problems are individually simple but collectively overwhelming: "How do I check part status?" "DVP is archived, can I still modify it?" "What's the EA completion rate?" "Where is the TKO process stuck?" These questions appear daily in Feishu groups and DMs, with answers often requiring cross-system investigation — reading code, querying databases, and flipping through manuals.

### The Insight

I noticed something interesting: as a product manager, I was already spending time writing small AI tools to help myself — a SQL query script, a manual-reading prompt, a permission debugging workflow. These were individual Skills, but I realized they could be **orchestrated into a complete Agent**. I mapped out **5 high-frequency scenarios**, trained **4 reusable Skills**, and built the IDP Agent from scratch.

## What I Did

### 4 Reusable Skills

| Scenario | Skill | Core Capability |
|---|---|---|
| System Debug | idp-debug | Agent follows a 6-step protocol to diagnose issues, tracing from frontend → backend → data layer |
| Data Query | NL Query | Natural language queries against the data warehouse — parts, workbenches, DVP/EA status |
| Operation Guide | Expert Skill | 8 operation manuals embedded, generates step-by-step guides with Feishu doc links |
| Data Dashboard | Analytics Skill | Auto-generates data analysis dashboards from frontend telemetry data |
| SOP Automation | Table Processing | Complex Feishu spreadsheet SOPs encapsulated as reusable Skills |

### Technical Architecture

- **Feishu Entry**: Private chat, group @mention, and topic threads — zero-friction access for all users
- **Agent Framework**: Built on Claude Code Agent SDK, supporting 20-turn multi-turn conversations, screenshot understanding, and file delivery
- **Knowledge Base**: 8 Markdown operation manuals + complete data warehouse schema documentation + business source code submodule (GitLab)
- **Data Access**: DataWorks read-only token, Agent queries the data warehouse directly
- **Deployment**: Matrix (Pre/Prod) + Docker + CI/CD automated pipeline
- **Observability**: OneTrack analytics + Shujing BI dashboard — monitoring request volume, success rate, latency, and token consumption
- **CLI Extension**: **106 CLI commands** developed, encapsulating core platform capabilities as command-line tools

### What I Innovated

**From Personal Tool to Replicable Framework.** The proudest outcome isn't the IDP Agent itself — it's the "Copy to System X" operational guide I wrote (14 steps). Any business team can swap Skills, knowledge bases, source code, and data sources to quickly build an Agent for their own system. This makes IDP Agent not just a product feature, but a **reusable enterprise AI Agent framework**.

**Skill-first Agent Design.** Instead of traditional RAG or fine-tuning, I chose a Skill-based architecture: each business capability is encapsulated as an independent Skill, and the Agent routes to the right Skill based on user intent. Adding a new scenario only requires writing a new Skill — no changes to the Agent core.

**Product Manager Trains the Skill.** The traditional workflow is: engineers write code, PMs write requirements. But in this project, I — as a product manager — independently trained the debug Skill and integrated it into the Feishu bot. This broke the assumption that "AI must be built by engineers." **Product managers can train Agents too, and they understand the business context better.**

**Human-AI Collaboration Boundaries.** I explicitly designed what the Agent "cannot do" — it cannot modify data, cannot operate the system on behalf of users, and complex business logic is redirected to the domain expert. Every response includes an AI disclaimer, ensuring users understand it's an assistant, not a decision-maker.

</En>

<Zh>

### 我的创新

**从个人工具到可复制框架。** 最让我自豪的不是 IDP Agent 本身，而是我把整个架构抽象成了一套「复制到系统 X」的实操指南（14 步）。任何业务团队只要替换 Skill、知识库、源码和数据源，就能快速搭建面向自己系统的 Agent。这意味着 IDP Agent 不只是一个产品功能，而是一个**可复用的企业 AI Agent 框架**。

**Skill-first 的 Agent 设计思路。** 不同于传统的 RAG 或微调方案，我选择了 Skill-based 架构：每个业务能力封装为独立的 Skill，Agent 根据用户意图自动路由到对应 Skill。扩展一个新场景只需写一个 Skill，不需要改 Agent 核心代码。

**产品经理训练 Skill 的范式突破。** 传统思路是研发写代码、产品写需求。但在这个项目中，我作为产品经理独立训练了 debug Skill，直接接入飞书机器人实现智能问答。这打破了「AI 必须由研发来做」的惯性认知——**产品经理也可以训练 Agent，而且更懂业务场景**。

**人机协作的边界设计。** Agent 不是万能的。我明确设计了它「不能做」的事——不能改数据、不能代替操作、复杂业务逻辑引导咨询流程专家。每条回答都附带 AI 免责声明，确保用户知道这是辅助工具而非决策者。

- **106 CLI commands** — covering core platform capabilities

## Reflection

This project changed how I think about product management. The traditional PM role is "write requirements, hand off to engineering." But with AI tools, a PM can now **build the product themselves** — not just spec it.

The key lesson: start from your own pain points. I didn't design an AI product in a vacuum. I built tools to solve my daily problems first, then generalized them into an Agent. This bottom-up approach is more likely to produce something people actually use.

The framework I built — the "Copy to System X" guide — is perhaps the most lasting contribution. It means every team in the organization can now build their own AI Agent in days, not months. That's the real leverage of AI in the enterprise.
